- Read in full: yes for save_manager.gd, actor.gd, persistence-notes.md, and all serialize/deserialize method bodies of the entity contracts. Large files (hero 1203L, level 1033L, char 607L, mob 552L) read in the serialize/deserialize + buff regions in full; non-persistence gameplay code in those files skimmed by grep, not line-by-line.
- Verdict: needs-hardening — the object graph (de)serialization contract is actually solid and symmetric, but the durability layer around it (when we save, how safely we write) is thin and there is ~200 lines of dead, out-of-sync duplicate serialization code that is a corruption trap.

## Improvements
- [P1] No autosave: `save_full_game()` is called from exactly ONE place — `wnd_game.gd:89` `_on_save_quit()` (manual "Save & Quit"). — A crash, OS kill, mobile background, or (given the explicit `user://` HTML5 target) a closed browser tab loses the entire run since the last manual save; `GameManager.descend()` (game_manager.gd:146) does not save. — Autosave on every floor transition (descend/ascend) and on `NOTIFICATION_WM_CLOSE_REQUEST` / `NOTIFICATION_APPLICATION_PAUSED`; SPD saves on every level change. This is the single highest-value fix.
- [P1] Non-atomic write with no backup: `save_full_game()` opens `SAVE_PATH` `WRITE` and `store_var` straight over the only save file (save_manager.gd:45-51). — A crash/power-loss mid-`store_var` truncates or corrupts the sole save with no recovery path — exactly the truncation-breakage class this repo is prone to. — Write to `SAVE_PATH + ".tmp"`, `flush()`/`close()`, then `DirAccess.rename_absolute()` over the real file, and rotate a `.bak`; optionally `OS.set_use_file_access_save_and_swap(true)`. Also verify the file after write (currently only the `open()` null is checked, not the store result).
- [P2] Dead, out-of-sync duplicate serialization (~200 lines): `_serialize_mobs/_deserialize_mobs`, `_serialize_heaps/_deserialize_heaps`, `_serialize_traps`, `_serialize_plants`, `_serialize_blobs` and the `_create_entity_by_type` / `_find_script_for_type` / `_pascal_to_snake` helpers (save_manager.gd:309-537) are never called — `Level.serialize()` (level.gd:717) owns the real contract. — They encode a DIFFERENT, wrong contract: `_serialize_heaps` keeps the live `Item` object inside the heap dict, and the reconstruction path keys off a `"type"` PascalCase field that no entity actually writes (mobs write `"_class"`+`"mob_id"`, traps/plants write `"_script_path"`). A future dev wiring these up would silently corrupt saves. — Delete the dead block; keep serialization single-sourced in the entity `serialize()`/`deserialize()` methods.
- [P2] No migration path despite a version field: `SAVE_VERSION` (=1) only rejects *newer* saves (save_manager.gd:80-83); there is no upgrade path for *older* saves. — As the schema drifts, old saves load "successfully" but silently default every renamed/added field (every `data.get(key, default)`), so a stale save quietly loads a subtly-wrong hero/level. — Add a `_migrate(save, from_version)` dispatch and bump `SAVE_VERSION` whenever a contract changes; refuse-or-migrate rather than best-effort default.
- [P3] `RegularLevel.serialize()` only calls `super.serialize()` (regular_level.gd:531-535) — room list / room types are not persisted. — Terrain map survives (tiles are saved), but any runtime logic that reads `level.rooms` after a continue (room-scoped respawns, shop/special-room detection, respawn cell selection) sees an empty room set. — Verify whether `rooms` is read post-generation; if so, persist a compact room descriptor. If rooms are purely generation scaffolding, document that they are intentionally dropped.

## Optimizations
- [P3] `_deserialize_game_manager` assigns `GameManager.stats = data.get("stats", {})` (save_manager.gd:175) — stores the raw dict returned by `get_var` without `duplicate(true)` (the serialize side does duplicate). Low impact, but inconsistent with the write path and shares references with the just-loaded dict. — Duplicate on the way in for symmetry.
- [P3] `store_var(data, true)` is fine and fast, but the whole `data` dict is already pre-flattened to plain dicts/primitives before write, so `full_objects=true` is only needed for nested typed dicts (level_cache `Dictionary[int,Dictionary]`). No action needed; noted so a future refactor doesn't "optimize" the flag away and break typed-key round-tripping.

## Additions
- [P1/value] Autosave-on-transition + save-slot backup rotation (SPD ref): SPD writes `game.dat` on every level change and keeps per-hero-slot saves; adopting even single-slot autosave + one `.bak` would move this from "manual save or lose it" to durable. Pairs with the P1 findings above.
- [P2/value] Per-record schema version (SPD Bundle/Bundlable ref): SPD's `Bundle` tags every bundled object with a class id AND supports per-class migration on load. Here `_script_path` is the class-id equivalent (good, and used consistently for buffs/traps/plants/mobs), but there is no per-object version, so targeted field migrations are impossible. Add an optional `_v` per serialized record for future-proof migration.
- [P3/value] Integrity check: beyond the null/`is Dictionary` guard, a stored length or checksum would let load distinguish "no save" from "corrupt/partial save" and surface a clean error instead of a defaulted-field silent load.

## Save/load & coupling notes
- Contract state: the object-graph contract is **complete and symmetric** for the walked types. `Actor` (actor_id/pos/active) → `Char` buffs via `_serialize_buffs` → `Hero`/`Mob` extend cleanly with `super`; `Item` and all checked subclasses (weapon/armor/ring/wand) call `super.serialize()`; buff subclasses (burning/bleeding/regeneration/hunger/combo) all call `super.serialize()` so the load-critical `_script_path` is never dropped. Reconstruction keys are consistent: buffs/traps/plants/heaps/level use `_script_path` + `load()`; mobs use `mob_id` + `MobFactory`; items use `item_id` + `Generator`, then `deserialize()` overlays state. `resolve_post_load` hooks re-link mobs (level.gd:823) and equipped artifacts (hero.gd:1037). Heaps correctly convert live `item` → `item_data` on save and back on load (level.gd:729-737, 796-810).
- Actor reactivation: `deserialize_actor` restores the `active` *flag* but does NOT re-register with TurnManager. Reactivation is instead done in `loading_scene._transition_to_game()` (loading_scene.gd:378-399), which `clear_actors()` then re-`activate()`s hero(es) + mobs. The continue flow (`title_scene._on_continue_pressed` → `load_full_game` → LoadingScene) routes through this, so turn state is correctly rebuilt. Note the coupling: correctness of load depends on always entering via LoadingScene; a code path that calls `load_full_game()` without it would leave every actor deserialized-but-unregistered (frozen turn system).
- Load ordering is deliberate and load-bearing: GameManager → QuestHandler (before NPCs re-register) → level_cache → current_level → heroes → reattach `hero.level`. `actor_id` `_next_id` high-water-mark is maintained on load (actor.gd:63-64) so post-load spawns don't collide.
- Minor asymmetry: `Hero.serialize` omits transient `shielding`/`invisible`/`paralysed` (Mob saves all three). Buffs re-apply most of these on load, so low risk, but a save taken mid-shield loses the shield.
- Autoload dependencies (tight but expected for this architecture): GameManager, TurnManager, QuestHandler, ItemAppearance, EventBus, LevelFactory, Generator, MobFactory (via level), AudioManager, ItemCatalog. SaveManager reaches directly into GameManager internals (`_level_cache`, `heroes`, `current_level`) — level cache is stored as already-serialized `Dictionary[int,Dictionary]` (game_manager.gd:178 `_level_cache[depth] = current_level.serialize()`), so no live Level nodes leak into the save.

## Research notes
- Godot binary saves are not crash/power-safe by default: FileAccess supports temp-file + swap, and `OS.set_use_file_access_save_and_swap(true)` enables it, but the docs are explicit it protects against Godot-level errors, not OS power loss (buffers may be uncommitted before rename). Manual temp→rename + `.bak` rotation is the standard mitigation. Refs: [FileAccess docs](https://docs.godotengine.org/en/stable/classes/class_fileaccess.html), [GDQuest save systems](https://www.gdquest.com/library/save_game_godot4/), [godot#98360 power-loss corruption](https://github.com/godotengine/godot/issues/98360).
- `store_var`/binary is the correct choice here over JSON (preserves int dict keys — matters for `_level_cache` and trap/plant `pos` keys, which JSON would stringify) and over ConfigFile (which is for flat settings, not object graphs). Settings/rankings using separate flat `store_var` files is reasonable.
- SPD fidelity reference: SPD's `Bundle`/`Bundlable` tags each object with a class id + supports per-class load-time migration, and the game autosaves on every level transition per hero slot. This port matches the class-id idea via `_script_path`/`*_id` factories (good), but is missing (a) autosave-on-transition, (b) per-record versioning/migration, and (c) atomic write + backup — the three items that make SPD's persistence durable in practice.
