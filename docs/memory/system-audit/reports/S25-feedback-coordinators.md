# Feedback coordinators — Audit

- Files: `src/mechanics/scene_feedback_coordinator.gd`, `src/mechanics/scene_visual_coordinator.gd`, `src/mechanics/environment_feedback_coordinator.gd`
- Read in full: yes (all three). Cross-read the delegating wrappers + call sites in `src/scenes/game_scene.gd` (truncated — read-only).
- Verdict: **needs-hardening** — the feedback layer is clean and SPD-faithful in *what* it shows, but it repaints the whole view on every mob action (per-turn cost is O(mobs) full FOV+visibility rebuilds), spawns this-turn sprites after visibility is applied (items flash in fog for one turn), and leaves a ~47-line dead inline block behind the extraction.

## Improvements
- [P2] **This-turn sprites render un-gated for one turn.** In `refresh_after_turn` the order is `_update_entity_visibility()` (scene_feedback_coordinator.gd:16) *then* `_refresh_item_sprites`/`_refresh_plant_sprites`/`_refresh_armed_bomb_sprites` (lines 20–22). New sprites are added to `_entity_layer` with default `visible = true` and only get FOV-gated on the *next* refresh, so an item/plant/bomb that appears in a fogged cell this turn is briefly visible through the fog. — Fix: refresh sprites first, then update visibility; or set initial `visible` from `level.visible[pos] or level.visited[pos]` at spawn in scene_visual_coordinator.gd:66–71 / :100–103 / :129–134.
- [P2] **Stale item icon when a heap's top item changes.** `refresh_item_sprites` keys sprites by cell `pos` only and skips any pos already in `_item_sprites` (scene_visual_coordinator.gd:62–65), so when the top `item` of an existing heap changes (a different item dropped onto the pile, heap type change) the old icon persists until the heap empties. — Fix: cache the item identity per pos and re-`setup_from_item` when it differs.
- [P3] **Missing `is_instance_valid` guard on item/plant/bomb visibility loops.** `update_entity_visibility` guards mob sprites with `is_instance_valid(sprite.character)` (scene_visual_coordinator.gd:187) but writes `sprite.visible = …` directly for item/plant/bomb sprites (lines 199–210). A freed sprite lingering in a dict would error; the refresh functions normally keep the dicts clean, so this is defensive only.

## Optimizations
- [P1] **Full FOV + visibility rebuild on every mob action.** `on_mob_action` recomputes `update_fov` + `fog_of_war.update_visibility()` + full `_update_entity_visibility()` (which itself rebuilds hero identifiers *and* blob overlays and scans every mob/item/plant/bomb sprite) for *each* acting mob (scene_feedback_coordinator.gd:41–47). With N mobs stepping per hero turn that is N full visibility rebuilds; SPD computes FOV once per hero observe (`Dungeon.observe()` → `GameScene.afterObserve()`), not per-mob. — Fix: coalesce visibility refresh to once per turn (a dirty flag drained in `refresh_after_turn`), and/or gate the per-action refresh on the acting mob being visible / near the hero. Biggest perf win in this system.
- [P2] **Redundant double blob-overlay rebuild.** Refresh paths call `_refresh_blob_overlays()` and then `_update_entity_visibility()`, and the latter calls `refresh_blob_overlays` internally (scene_visual_coordinator.gd:211) — e.g. game_scene.gd:634 then :640, and the `refresh_after_turn` path. Blob cells are recomputed and re-pushed to `_blob_layer` twice per refresh. — Fix: drop the standalone `_refresh_blob_overlays()` when `_update_entity_visibility()` follows, or remove the internal call and always drive it explicitly.
- [P2] **~47 lines of dead inline code behind the extraction.** `_update_entity_visibility()` in game_scene.gd delegates to the coordinator then `return`s at line 1113; lines 1114–1160 are the old inline implementation, now unreachable. Same debt pattern S23 flagged in this file. Backlog-only — game_scene.gd is in `TRUNCATED_FILES.txt`, so no auto-edit.

## Additions
- [P2] **No tests + untestable shape.** All three are static funcs taking an opaque `scene: Variant` and duck-typing `scene._*` / `has_method` calls — brittle and impossible to unit-test in isolation. Extract a small typed feedback target (or pass the concrete typed nodes: level, fog, tile_map, effect_manager, camera) so behavior can be covered and the `Variant`/`has_method` guards can drop.
- [P3] **Hard autoload coupling.** The coordinators reach `GameManager`, `AudioManager`, `NetworkManager`, `OnlineEventCodec`, `ConstantsData` directly. For framework extraction, inject these (constructor/param) so the feedback layer is reusable outside this game's autoload set.
- [P3] **Duplicate status-feedback paths.** `on_hero_damaged_detailed` surfaces buff status VFX directly (scene_feedback_coordinator.gd:91–92) while `on_status_effect_applied` also emits status feedback *and* a world event (lines 101–106); the same buff-damage event can produce double VFX / inconsistent netcode depending on which fires. Unify status-feedback emission through one path.

## Save/load & coupling notes
- Pure transient view layer — nothing here is serialized and nothing needs to be; correct. No persistence contract to harden.
- Structural debt is coupling, not durability: static funcs over an opaque `scene` Variant with duck-typed member access + direct autoload dependencies. The per-mob-action repaint is the only *behavioral* hot spot.

## Research notes
- SPD drives FOV/visibility once per hero observe: `Dungeon.observe()` recomputes the field of view and `GameScene.afterObserve()` syncs sprite visibility, invoked from the actor loop when the hero acts — not once per mob action. This port instead refreshes on every `on_mob_action`, the root of the P1 optimization. Ref: [00-Evan/shattered-pixel-dungeon](https://github.com/00-Evan/shattered-pixel-dungeon) (`GameScene`, `Dungeon.observe`).
- Web search for the specific `afterObserve`/`observe` methods surfaced only general project pages ([itch devlog](https://shattered-pixel.itch.io/shattered-pixel-dungeon/devlog), [releases](https://github.com/00-Evan/shattered-pixel-dungeon/releases)); the architecture point is from the known source layout rather than the search.
