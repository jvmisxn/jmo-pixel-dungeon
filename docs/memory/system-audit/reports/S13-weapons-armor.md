# Weapons / armor / glyphs / enchants — Audit

- Files: `src/items/weapons/weapon.gd`, `melee_weapon.gd`, `missile_weapon.gd`, `spirit_bow.gd`, `weapon_enchantment.gd`; `src/items/armor/armor.gd`, `armor_glyph.gd`
- Read in full: yes (all 7 files)
- Verdict: **needs-hardening** — core damage/DR/STR/augment math is faithful and clean, but per-weapon differentiation is collapsed (uniform damage + uniform speed within a tier), SpiritBow loses all state on load, random loot never gets glyphs, and several enchant/glyph procs are inert or un-dispatched.

## Improvements
- [P1] **SpiritBow.deserialize() is `pass`** (`spirit_bow.gd:143`) — it discards the entire saved payload. Belongings restores it via `Generator.create_item("spirit_bow")` → `item.deserialize(data)` (`belongings.gd:387`), so on every load the bow's `level` (upgrades), augment, and enchantment reset to 0/none. `serialize()` even writes the data that `deserialize` throws away. Data-loss. Fix: `super.deserialize(data)` (drop the no-op override entirely, or call super then read `is_spirit_bow`).
- [P1] **All same-tier weapons are mechanically identical except reach.** `delay_factor` defaults to 1.0 and is never set in any `MeleeWeapon.create` / `MissileWeapon.create` branch (only serialized), so every weapon attacks at the same speed. Damage comes solely from `Weapon.get_damage_range()` by `tier`, so e.g. dagger, rapier, cudgel, gloves (all tier 1) have identical damage AND speed. SPD differentiates each weapon within a tier by DLY/damage/ACC/STR trade-offs. Fix: set per-weapon `delay_factor` (and ideally min/max overrides) in the factories.
- [P1] **Random armor never spawns with glyphs.** `Armor.random()` (`armor.gd:357`) stubs both the cursed-glyph and good-glyph branches with comments/`pass` (`:371`, `:373-374`), so loot armor is always glyph-less. Glyphs are only reachable via `inscribe()` (Scroll of Enchantment path, `upgrade_armor(true)`). Weapon side is fine — `Weapon.random()` does enchant at ≥0.85. Fix: apply `ArmorGlyph.random()` on the good roll (and a curse glyph once curse glyphs exist).
- [P2] **Curse weapon-enchant procs are never dispatched.** `WeaponEnchantment.proc()` (`:32`) only matches the 13 non-curse ids; the implemented `_annoying_proc`/`_sacrificial_proc`/`_displacing_proc` are dead. Compounded by `random_curse()` having no callers, so curse enchantments are also never generated — the whole curse-enchant layer is inert. Fix: dispatch curse ids in `proc()` and wire `random_curse()` into cursed-weapon generation.
- [P2] **FlowGlyph & EntanglementGlyph apply inert buffs.** Both build a raw `Buff.new()` and set `buff_id = "Haste"`/`"Rooted"` by hand (`armor_glyph.gd:463-466`, `:482-485`) instead of `Haste.new()` / `Rooted.new()` (which `SwiftnessGlyph` does correctly). A base `Buff` has none of the subclass speed/root behavior, so Flow's on-proc haste and Entanglement's root do nothing. Fix: instantiate the real classes.
- [P2] **`has_curse_glyph()` is always false.** `armor.gd:299` reads `glyph.get("is_curse")`, but `ArmorGlyph` has no `is_curse` property (unlike `WeaponEnchantment`). So the curse-glyph upgrade-removal branch (`armor.gd:240`) never fires and `has_good_glyph()` is always true when a glyph exists. Fix: add `is_curse` to `ArmorGlyph` (defaulting false) once curse glyphs are added.
- [P3] **Displayed STR req goes stale after upgrades.** `str_requirement` is cached at `create()` with level 0; `get_stats_text()` (`weapon.gd:238`, `armor.gd:381`) prints the cached value. Armor recomputes on upgrade via `_update_str_requirement()`, but Weapon has no equivalent, so an upgraded weapon shows its unreduced STR req. Combat/accuracy already use the live `get_str_requirement()`, so this is display-only. Fix: refresh weapon `str_requirement` in `upgrade_weapon()`.

## Optimizations
- [P3] **`MeleeWeapon.value()`** (`melee_weapon.gd:194`) is a no-op override — body is exactly `return super.value()`. Remove. *(auto-fixed this run)*
- [P3] **`MissileWeapon.get_damage_range()`** (`missile_weapon.gd:239`) is a no-op override that fetches `super.get_damage_range()` and returns it unchanged, with a comment claiming reduced missile damage that the code never applies. Remove the override (and misleading comment). *(auto-fixed this run)*
- [P3] `_blazing_proc` `ClassDB.class_exists("Burning")` / `ClassDB.instantiate(...)` (`weapon_enchantment.gd:220-222`) can never succeed — `Burning` is a GDScript `class_name`, not a native ClassDB class — so the branch always falls through to the script-load fallback. Simplify to `Burning.new()` (the class is a global) and drop the dead ClassDB branch.

## Additions
- [P2] Several enchants are simplified stand-ins vs SPD: `_shocking_proc` has no chain-lightning to adjacent enemies (flat bonus only, `:249`), `_kinetic_proc` doesn't store overkill damage (random 20%/+50% instead, `:336`), `_grim_proc` uses a bespoke <20%-HP curve. Worth restoring true behavior for parity.
- [P2] `MissileWeapon` uses the melee damage formula (`get_damage_range` delegates to `Weapon`); SPD missiles have their own lower per-weapon damage bands. Give missiles real per-id damage.
- [P3] No curse **armor glyphs** exist (SPD: Anti-Entropy, Corrosion, Displacement, Metabolism, Multiplicity, Stench, Overgrowth, Bulk). Add them so cursed armor has negative glyphs.
- [P3] No unit coverage for the pure math here: `get_damage_range`/`dr_max`/`dr_min`/`dr_roll`/`get_str_requirement`/augment factors and the enchant/glyph serialize→create→deserialize round-trip — clean first framework-extraction test targets.

## Save/load & coupling notes
- Weapon/Armor serialize the full contract (tier, augment, hardened/curse-infusion/mastery flags, id-progress, nested enchantment/glyph) and rebuild nested objects through their factories on load — solid, **except** SpiritBow (P1 above).
- Enchantments/glyphs serialize by id and are re-created via `WeaponEnchantment.create` / `ArmorGlyph.create` before `deserialize`, so color/name come from the factory — correct.
- Coupling: procs reach through duck-typed autoloads (`MessageLog`, `EventBus`, `DiscoveryCatalog`, `ConstantsData`) and `defender.level`/`belongings`; all null-guarded. Combat wiring is live — `hero.gd:521,951,963` call `proc_enchantment`/`proc_glyph`, `char.gd:119`/`hero.gd:1157` call `dr_roll`.

## Research notes
- SPD `Weapon.java`: augment SPEED = ×0.7 dmg / ×0.667 DLY, DAMAGE = ×1.667 dmg / ×1.5 DLY — the port's constants (`weapon.gd:26-29`) match exactly, and the wiki confirms the same figures.
- SPD MeleeWeapon STR-req = `8 + tier*2 - triangular(lvl)` and damage `min = tier+lvl`, `max = (tier+1)*(5+lvl)` — the port matches; the gap is that SPD then per-weapon overrides min/max/DLY/RCH/ACC, which this port does not (P1).
