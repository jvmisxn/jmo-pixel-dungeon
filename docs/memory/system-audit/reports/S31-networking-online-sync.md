# Networking & online sync — Audit

- Files: `src/autoloads/network_manager.gd` (709), `src/mechanics/online_sync_coordinator.gd` (60), `src/mechanics/online_event_codec.gd` (714), `src/mechanics/online_action_codec.gd` (109), `src/mechanics/online_snapshot_utils.gd` (121)
- Read in full: yes (all 5). Consumer `src/scenes/game_scene.gd` read partially for wiring (truncated file, not edited).
- Verdict: **needs-hardening** — clean host-authoritative transport with a well-factored codec/utils/coordinator split, but sync is a per-turn full-state broadcast, the client re-derives the run config instead of trusting the host, and a lobby ready-toggle wipes the host's chosen icon.

## Improvements
- [P1] **Client re-sanitizes the host's authoritative run config.** `_client_receive_online_run_start(config)` calls `_sanitize_run_config(config)` again on the client (`network_manager.gd:671-672`). `_sanitize_run_config` rebuilds `party`/`player_infos` from the *client's own* `lobby_players` (`:561-588`), not from the host-provided config. If a client's lobby snapshot is even momentarily stale or reordered at run-start, host and client derive different party composition / class-to-slot mapping → hero/class desync for the whole run. Direction: clients should consume the host config verbatim (only fill defaults for genuinely-missing keys); reserve `_sanitize_run_config` for the host in `start_online_run`.
- [P2] **`set_local_ready` resets the host's profile icon to "warrior".** `network_manager.gd:317` is the only `_upsert_lobby_player` call site that omits `local_profile_icon_id`, so `profile_icon_id` falls back to the `"warrior"` default. Every time the host toggles ready in the lobby their chosen icon is silently overwritten and re-broadcast. All sibling host-side upserts (`:194,:295,:308`) pass `local_profile_icon_id`. Behavioral fix — add the missing argument. (Cosmetic/host-only, no data loss → P2.)
- [P3] **Remote mob moves are dropped before the mob sprite exists.** `apply_remote_mob_move` returns `false` and no-ops if the client has no sprite for that `actor_id` yet (`online_event_codec.gd:560-561`); the mob's position is only corrected by the next full snapshot. Low impact given snapshots cover it, but causes brief teleport-on-reveal.
- [P3] **Drop message logs even on failed deserialize.** `handle_item_dropped_world_event` logs `"%s drops %s"` whenever `drop_pos >= 0`, even if the item failed to deserialize (`dropped_item == null`, no heap created) — `online_event_codec.gd:603-624`. Cosmetic log/state mismatch.

## Optimizations
- [P1] **Per-turn full-state snapshot broadcast.** `OnlineSyncCoordinator.sync_snapshot` (`:4-9`) fires whenever `TurnManager.waiting_for_input` and RPCs `_serialize_online_snapshot()` — which is the *entire* `level.serialize()` (full `map` array + every mob + every heap/item) plus every hero's `serialize()` (`online_snapshot_utils.gd:4-17`) — reliably to all clients on every turn boundary. No delta/dirty-tracking. Bandwidth and CPU scale with level size × turn frequency; large `reliable` payloads can head-of-line stall ENet. Direction: diff against the last sent snapshot (the `capture_*_snapshot_state` helpers already compute per-actor pos/hp/action deltas — reuse them to send only changes), and reserve full snapshots for level transitions / late-join catch-up (`_server_register_player:604-607`).
- [P3] **`NetworkManager` is a God-object (30+ public methods; gdlint `max-public-methods`).** Mixes lobby membership, ENet transport/RPC, run-config sanitation, and prefs persistence in one autoload. Framework-extraction: split a `LobbyState` value object from the transport layer (mirrors the already-clean codec/utils extraction).

## Additions
- [P2] **No tests on the codec layer.** `OnlineActionCodec`, `OnlineEventCodec`, `OnlineSnapshotUtils` are pure `static`/`RefCounted` with no engine deps — ideal for round-trip unit tests: `make_item_ref`→`resolve_item_ref`, `encode_network_action`→`normalize_action_for_hero`, `serialize_shop_items`→`deserialize_shop_items`, `make_item_from_data`. Would lock down the serialization contract the whole online path depends on.
- [P3] **No mid-run online persistence.** NetworkManager persists only lobby prefs (`_save_prefs`, name/port/class/icon). `active_run_config`/`latest_run_snapshot` are runtime-only and cleared on `close_session` (`:119-121`). A host save+reload mid-online-run drops the session with no resume. Acceptable for live co-op, but document the limitation (host disconnect = run over).

## Save/load & coupling notes
- Persistence contract: prefs only (`user://network_prefs.cfg`); no online run state is serialized to the save file. Offline saves are unaffected.
- Coupling: reaches `GameManager` (heroes, depth, MAX_PARTY_SIZE), `PlayerProfile` (name/icon), `ConstantsData`, `TurnManager` (waiting_for_input), plus the codec trio. The scene interface is duck-typed via `Callable`s built in `OnlineSyncCoordinator._build_world_event_context` — flexible but unchecked (a renamed `game_scene` method fails silently at call time). Host-authoritative model is sound: `_server_receive_online_action` validates slot ownership before applying (`:701-709`).

## Research notes
- Godot 4.5 high-level multiplayer: `@rpc("any_peer"/"authority", "reliable")` used throughout — correct authority split (clients `rpc_id(1, ...)` to host; host broadcasts `authority`). Reliable is right for lobby/config but costly for the large per-turn snapshot (see P1 opt).
- SPD reference: base Shattered PD is single-player; this is an original co-op layer. The host-authoritative + per-slot-ownership action model is the sound choice; the gap vs production netcode is delta compression and trusting server-authoritative config on clients.
- Verified all `OnlineSnapshotUtils` symbols are live (25 refs to `find_hero_by_actor_id`, etc.); `serialize_shop_items` is used internally by `send_shop_refresh`. No dead code found in the online layer.
